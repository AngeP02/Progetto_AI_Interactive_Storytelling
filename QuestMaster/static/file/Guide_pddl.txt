Advanced PDDL Guide for LLMs: Syntactically and Semantically Correct GenerationThis guide is designed to teach you how to generate correct PDDL (Planning Domain Definition Language) code. PDDL is a standardized, LISP-based language for describing AI planning problems. It is crucial to pay attention to syntax (parentheses, keywords) and semantics (logical consistency and adherence to definitions).PDDL is structured into two main files:domain.pddl: Defines the environment, object types, predicates (properties), and possible actions.problem.pddl: Defines a specific problem instance, with concrete objects, the initial state, and the goal to achieve.1. The domain.pddl File – The Rules of the WorldThis file defines the logical structure and dynamics of the planning world.1.1. Domain DefinitionAlways start with the domain definition.Syntax:Snippet di codice(define (domain <domain-name>)
  ; ... domain body ...
)
<domain-name>: A symbolic name for the domain (e.g., rover-world, blocksworld). It must exactly match the domain name specified in the problem.pddl file.Convention: Use lowercase letters and hyphens (-) to separate words.1.2. Requirements (:requirements)Specifies the PDDL features being used. Highly recommended for clarity and compatibility.Syntax:Snippet di codice(:requirements <...requirements...>)
Common Examples::strips: The basic core of PDDL (additive/deleterious effects). Essential.:typing: Allows defining types for objects (e.g., robot, location). Highly recommended.:negative-preconditions: Allows negated predicates in preconditions.:disjunctive-preconditions: Allows the use of (or ...) in preconditions.:equality: Allows the use of (= <obj1> <obj2>).:adl: (Action Description Language) Enables a richer set of features, including the preceding ones, plus quantifiers.Example:Snippet di codice(:requirements :strips :typing :negative-preconditions)
1.3. Types (:types)Defines a hierarchy of types for the objects. Required if :typing is used.Syntax:Snippet di codice(:types
  <type1> <type2> ... - <super-type_or_object>
  <type3> - <another-super-type>
  ...
)
Objects can have types (e.g., robot, room, ball).Types can be organized hierarchically (e.g., rover vehicle - mobile, probe camera - instrument).object is the default super-type if none is specified.Semantics: An object of a sub-type is also an instance of its super-type.Example:Snippet di codice(:types
  robot vehicle - mobile      ; robot and vehicle are subtypes of mobile
  room location - place       ; room and location are subtypes of place
  transportable-object
  mobile place - object       ; mobile and place are direct subtypes of object
)
1.4. Constants (:constants)Defines objects that exist in all problem instances for this domain.Syntax:Snippet di codice(:constants
  <constant1> - <type1>
  <constant2> <constant3> - <type2>
)
Useful for fixed elements of the world (e.g., a specific ground-station).Constants must be typed if :typing is active.Example:Snippet di codice(:constants
  earth - planet
  base-station - location
)
1.5. Predicates (:predicates)Define properties of objects or relationships between them, which can be true or false.Syntax:Snippet di codice(:predicates
  (<predicate-name1> ?<var1> - <type1> ?<var2> - <type2> ...)
  (<predicate-name2> ?<var3> - <type3>)
  ...
)
Each predicate has a name and a list of typed parameters.Variables are introduced by ? (e.g., ?r, ?loc).The type of each variable must be declared if :typing is used.Common Error: Using a predicate in actions, init, or goal without having declared it here.Semantics: A "grounded" predicate (with specific objects instead of variables) is a proposition that is true or false in a given world state.Example:Snippet di codice(:predicates
  (at ?rob - robot ?pos - room)  ; Robot ?rob is in room ?pos
  (connected ?r1 - room ?r2 - room) ; Room ?r1 is connected to ?r2
  (clean ?r - room)            ; Room ?r is clean
  (holding ?rob - robot ?obj - transportable-object) ; Robot ?rob is holding ?obj
)
1.6. Actions (:action)Define how the world state can change. This is the most complex part.Syntax:Snippet di codice(:action <action-name>
  :parameters (<?var1> - <type1> ?<var2> - <type2> ...)
  :precondition (and ; or a single predicate, or or/not if requirements allow
    (<predicate1> ... )
    (not (<predicate2> ... )) ; if :negative-preconditions is active
    ...
  )
  :effect (and ; or a single effect
    (<predicate_becomes_true> ...)
    (not (<predicate_becomes_false> ...))
    ...
  )
)
<action-name>: Unique name for the action.:parameters: Variables used in the action, with their types.Crucial Semantic Rule: All variables used in :precondition and :effect must be declared here in :parameters.:precondition: A logical condition that must be true for the action to be executed.Usually a conjunction (and ...).Can contain positive predicates and, if specified in :requirements, negative ones (not (...)).Common Error: Omitting (and ...) if there are multiple preconditions.:effect: Changes to the world state if the action is executed.Usually a conjunction (and ...).Effects are atoms (grounded predicates) that are added to the state (become true) or removed from the state (become false using (not (...))).Common Error: Omitting (and ...) if there are multiple effects.Action Example:Snippet di codice(:action move
  :parameters (?r - robot ?from - room ?to - room)
  :precondition (and
    (at ?r ?from)
    (connected ?from ?to)
  )
  :effect (and
    (not (at ?r ?from))  ; The robot is no longer in ?from
    (at ?r ?to)          ; The robot is now in ?to
  )
)

(:action clean-room
  :parameters (?r - robot ?s - room)
  :precondition (and
    (at ?r ?s)
    (not (clean ?s)) ; Requires :negative-preconditions
  )
  :effect (and
    (clean ?s)
  )
)
Distinct Names: Actions and predicates must have distinct names. (move ...) is an action, (at ...) is a predicate.2. The problem.pddl File – The Game to PlayThis file describes a specific situation (instance) to be solved using the rules defined in domain.pddl.2.1. Problem DefinitionSyntax:Snippet di codice(define (problem <problem-name>)
  (:domain <matching-domain-name>)
  ; ... problem body ...
)
<problem-name>: A symbolic name for this specific instance (e.g., rover-problem1).(:domain <matching-domain-name>): Crucial. This name must exactly match the domain name defined in the domain.pddl file. This is the link between the two files.2.2. Objects (:objects)Lists all concrete objects that exist in this problem instance, specifying their type.Syntax:Snippet di codice(:objects
  <obj1> <obj2> - <typeA>
  <obj3> - <typeB>
  ...
)
The objects here are instances of the types defined in (:types) in the domain file.If :typing is not in the requirements, types are not specified. But using :typing is a best practice.Common Error: Using an object in :init or :goal without declaring it here, or assigning a type not declared in the domain.Example:Snippet di codice(:objects
  robby - robot
  kitchen living-room corridor - room
  apple - transportable-object
)
2.3. Initial State (:init)Defines the initial state of the world. It is a list of "grounded" predicates (instantiated with specific objects) that are true at the start.Syntax:Snippet di codice(:init
  (<grounded-predicate1> ...)
  (<grounded-predicate2> ...)
  ...
)
Each element is a predicate declared in the domain, but with specific objects (declared in :objects or :constants) replacing the variables.Only the true facts are listed. Everything not listed is implicitly false (Closed World Assumption).Common Error: Using a predicate not declared in the domain, or using objects not declared in :objects.Example:Snippet di codice(:init
  (at robby kitchen)
  (connected kitchen corridor)
  (connected corridor kitchen) ; Connections are often bidirectional
  (connected corridor living-room)
  (connected living-room corridor)
  ; (not (clean kitchen)) ; If explicitly stating negative facts here (less common)
  (clean living-room) ; If the living room is clean initially
)
*Note: It is more common to specify only positive predicates in :init. The absence of (clean kitchen) implies it is false.*

#### 2.4. Goal (:goal)
Defines the condition the planner must achieve. It is a logical condition on grounded predicates.
Syntax:
```pddl
(:goal (and ; or a single grounded-predicate
  (<goal-predicate1-grounded> ...)
  (<goal-predicate2-grounded> ...)
  ...
))
Usually a conjunction (and ...) of grounded predicates.When all predicates specified here become true, the problem is solved.Common Error: Omitting (and ...) if there are multiple goals, or using undeclared predicates/objects.Example:Snippet di codice(:goal (and
  (at robby living-room)
  (clean kitchen)
  (clean living-room)
))
Summary of Common Errors and Fundamental Syntactic/Semantic RulesDomain Name Consistency: (define (domain domain-name)) in domain.pddl must match (:domain domain-name) in problem.pddl.Predicate Declaration: Every predicate used in :init, :goal, or in an action's :precondition/:effect must be declared in the (:predicates) section of the domain.Object/Constant Declaration: Every object used in :init or :goal (or as an argument of an action in a plan) must be declared in (:objects) of the problem or (:constants) of the domain.Parameter/Object Typing:If :typing is used, all variables in predicate and action parameters must have a declared type.All objects in (:objects) and constants in (:constants) must have a declared type.The types used must be those declared in (:types).Action Variable Scope: All variables ?var used in the :precondition and :effect sections of an action must be defined in the :parameters section of that specific action. They are not global.Correct Format:Use (define (domain ...)) and (define (problem ...)).Respect keywords (e.g., :action, :parameters, :precondition, :effect, :types, :predicates, :objects, :init, :goal).Respect the S-expression structure (balanced parentheses).Unique Names: Actions and predicates must have distinct names. Types and predicates/actions can share the same name, but it is best to avoid it for clarity.Use of (and ...): If a precondition, effect, or goal has multiple clauses, they must be enclosed in an (and ...) block. A common mistake is listing them sequentially without and.Correct Effects: Effects either add facts (<predicate> ...) or remove facts (not (<predicate> ...)).No Complex Logic in Effects (Base STRIPS): In base PDDL (e.g., :strips), effects are just lists of atoms to add/remove. No or, forall, exists, or conditional effects (when) unless more advanced requirements are specified (e.g., :conditional-effects, :adl).Summary of Fundamental KeywordsKeywordFilePrimary SectionMain PurposedefineBothRootStarts the definition of the domain or the problem.domaindomain.pddlInside defineDeclares the name of the domain.problemproblem.pddlInside defineDeclares the name of the problem.:domainproblem.pddlInside defineSpecifies which domain the problem refers to.:requirementsdomain.pddlSpecifies the PDDL features used.:typesdomain.pddlDefines object types and their hierarchy.:constantsdomain.pddlDefines global, constant objects for the domain.:predicatesdomain.pddlDefines properties (true/false) and their signatures.:actiondomain.pddlStarts the definition of an action.:parametersdomain.pddlInside :actionDeclares variables (with types) used by an action.:preconditiondomain.pddlInside :actionConditions that must be true to execute the action.:effectdomain.pddlInside :actionChanges to the world state after the action.:objectsproblem.pddlLists the specific objects of the instance and their types.:initproblem.pddlDefines the initial state of the world (true facts).:goalproblem.pddlDefines the target state to reach.andBothPrecond/Effect/GoalLogical conjunction of clauses.notBothPrecond/EffectLogical negation of a clause.Note on the table: In a .txt file, table formatting might not be perfect, but the data is present.Advanced Tips for LLM GenerationPrioritize Syntactic Correctness: Parentheses must always balance. Keywords must be correct and in the right place.Respect Declarations: Never use a type, predicate, object, or constant that has not been previously declared in the appropriate section.Type Consistency: Ensure that the types of objects passed to predicates (in init, goal, precondition, effect) and the types of variables/parameters match the declarations of the predicates and actions.Initial Minimalism: Start with simple problems and minimal requirements (e.g., :strips, :typing). Add complexity gradually.Readability: Even as a machine, generating readable PDDL (indentation, clear names) aids in debugging and understanding. Use descriptive names for domains, problems, types, predicates, and actions.Basic Semantic Check:The preconditions of an action should logically enable the effects.Avoid actions with contradictory effects (e.g., adding and removing the same predicate simultaneously, unless this is a desired behavior managed by more complex semantics not covered here).Ensure the goal is potentially reachable given the actions and the initial state (this is harder to guarantee a priori but is a good principle).Attention to Unbound Variables: Every variable appearing in an action's precondition or effect MUST be among the parameters of that action. This is a very common mistake.Learn from Errors: When a PDDL planner reports an error in the generated code, analyze the error against this guide to understand the syntactic or semantic violation.By following these guidelines, you should be able to generate valid and meaningful PDDL files. Happy planning!
